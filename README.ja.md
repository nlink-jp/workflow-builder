# workflow-builder

自然言語のタスク記述から実行可能なシェルスクリプトを生成する、LLM 駆動のワークフロービルダー。構造化されたツールレジストリで利用可能な CLI ツールを理解し、データフローを計画し、コンテナ内での実行とフィードバックを通じてスクリプトを反復的に改善する。

**ステータス: 設計フェーズ** — アーキテクチャとツールレジストリは定義済み。実装は後日。

[English README is here](README.md)

## 基本思想

LLM エージェントは確率的である — 同じプロンプトでも毎回異なる結果を返す可能性がある。
これは、定期実行やミッションクリティカルなワークフローにおいて、LLM による直接的な
タスク実行を信頼できないものにしている。

workflow-builder は **LLM を実行時ではなくビルド時に使う** ことでこの問題を解決する。
LLM がシェルスクリプトを計画・生成・デバッグし、反復的なフィードバックを経て正しく
動作することが確認されたら、そのスクリプトは LLM なしで動作する決定的なワークフロー
となる — コンパイルされたコードと同じように。

> **自然言語からワークフローをコンパイルし、決定的に実行する。**

## コンセプト

```
「昨日のサイバーセキュリティニュースを    ──→  LLM がタスクを分析
  収集して、日本語に翻訳して、             ──→  ツールチェーンを計画
  #security-news にキュレーション          ──→  シェルスクリプトを生成
  ダイジェストを投稿して」                 ──→  コンテナ内で実行
                                          ──→  結果を検証
                                          ──→  動作するワークフローを出力
```

## コンポーネント

| コンポーネント | 説明 |
|---|---|
| **Tool Registry** (`tools/*.yaml`) | CLI ツールの構造化定義 — フラグ、I/O 形式、使用例、組み合わせヒント |
| **Container Image** (`container/`) | nlink-jp ツール群 + 標準ユーティリティをプリインストールしたサンドボックス実行環境 |
| **Orchestrator** | LLM 駆動パイプライン: 計画 → 生成 → 実行 → 評価 → 反復 |

## 仕組み

1. **定義** — YAML でツールの使い方を構造化記述
2. **記述** — タスクを自然言語で記述
3. **計画** — LLM がツールレジストリ + タスクを読み、Chain-of-Thought を生成
4. **生成** — LLM が計画をシェルスクリプトに変換
5. **実行** — コンテナ内でスクリプトを実行（サンドボックス）
6. **評価** — LLM が結果を分析、成功判定またはエラー診断
7. **反復** — 必要に応じて修正して再生成（最大 N 回）
8. **出力** — 動作するシェルスクリプト（再利用可能）
